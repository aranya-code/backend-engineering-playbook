# Introduction

Amazon Kinesis Data Streams is built around two primary components:

- Producers
- Consumers

A **Producer** continuously writes records into a Kinesis stream, while a **Consumer** reads and processes those records.

Unlike traditional messaging systems where a message is often consumed once, Amazon Kinesis allows **multiple independent consumers** to process the same stream simultaneously.

This enables real-time analytics, monitoring, machine learning, and event processing from a single data stream.

---

# Producer-Consumer Model

```
Producer

↓

Amazon Kinesis Data Stream

↓

Consumers
```

One producer can send millions of records.

Multiple consumers can process the same records independently.

---

# PutRecord

Writes a single record.

---

## AWS CLI Example

```bash
aws kinesis put-record \
    --stream-name OrderStream \
    --partition-key customer-101 \
    --data "Order Created"
```

---

## Boto3 Example

```python
import boto3

kinesis = boto3.client("kinesis")

kinesis.put_record(
    StreamName="OrderStream",
    Data="Order Created",
    PartitionKey="customer-101"
)
```

---

# PutRecords

Writes multiple records in one request.

```
Application

↓

10 Records

↓

PutRecords()

↓

Kinesis
```

Benefits

- Lower latency
- Fewer API calls
- Higher throughput
- Lower cost

---

## Boto3 Example

```python
response = kinesis.put_records(
    StreamName="OrderStream",
    Records=[
        {
            "Data": "Order 101",
            "PartitionKey": "customer-101"
        },
        {
            "Data": "Order 102",
            "PartitionKey": "customer-102"
        }
    ]
)
```

---

# Producer Best Practices

- Use PutRecords whenever possible.
- Choose good Partition Keys.
- Compress large payloads.
- Implement retry logic.
- Monitor throttling.

# Reading Records

```

↓

Shard Iterator

↓

↓

Process Records
```

---

## Get Shard Iterator

Before reading records,

the consumer requests a shard iterator.

```python
iterator = kinesis.get_shard_iterator(
    StreamName="OrderStream",
    ShardId="shardId-000000000000",
    ShardIteratorType="LATEST"
)
```

---

# Shard Iterator Types

Amazon Kinesis supports multiple iterator types.

| Iterator Type | Description |
|---------------|-------------|
| LATEST | Read new records |
| TRIM_HORIZON | Read from oldest available record |
| AT_SEQUENCE_NUMBER | Start at a specific sequence number |
| AFTER_SEQUENCE_NUMBER | Start after a sequence number |
| AT_TIMESTAMP | Start from a timestamp |

---

# GetRecords

Retrieve records from the stream.

```python
records = kinesis.get_records(
    ShardIterator=iterator["ShardIterator"]
)
```

# Consumer Types

Amazon Kinesis supports two consumer models.

- Shared Fan-Out Consumers
- Enhanced Fan-Out Consumers

---

# Shared Fan-Out

Multiple consumers share shard read throughput.

```
Shard

↓

Consumer A

↓

Consumer B

↓

Consumer C
```

Characteristics

- Shared bandwidth
- Lower cost
- Higher latency under heavy load

---

# Enhanced Fan-Out

Each consumer receives dedicated throughput.

```
Shard

↓

Consumer A

Dedicated Throughput

-------------------

Consumer B

Dedicated Throughput
```

Benefits

- Lower latency
- Faster processing
- Independent throughput

Ideal for production systems with multiple consumers.

---

# Consumer Lag

Consumer Lag measures how far behind a consumer is compared to the latest record.

```
Latest Record

↓

Consumer

↓

Lag
```

High lag indicates the consumer cannot keep up with incoming traffic.

---

# Checkpointing

Consumers track processed records using checkpoints.

```
Read Record

↓

Process Record

↓

Save Checkpoint
```

Checkpointing prevents duplicate processing after failures.

---

# Retry Behavior

Temporary failures should trigger retries.

```
Read Record

↓

Processing Failed

↓

Retry

↓

Success
```

Applications should use exponential backoff.

---

# Error Handling

Typical processing workflow.

```
Receive Record

↓

Validate

↓

Process

↓

Store

↓

Checkpoint
```

Invalid records should be logged and isolated.

---

# Shared Fan-Out vs Enhanced Fan-Out

| Feature | Shared Fan-Out | Enhanced Fan-Out |
|----------|----------------|------------------|
| Throughput | Shared | Dedicated |
| Latency | Higher | Lower |
| Cost | Lower | Higher |
| Consumer Isolation | No | Yes |
| Best For | Small workloads | High-throughput production systems |

---

# Best Practices

- Batch writes using PutRecords.
- Choose high-cardinality Partition Keys.
- Use Enhanced Fan-Out for latency-sensitive applications.
- Implement checkpointing.
- Handle retries gracefully.
- Monitor consumer lag.
- Scale shards before throughput limits are reached.
- Process records asynchronously where appropriate.

---

# Common Mistakes

## Writing One Record Per Request

Batch records using PutRecords to improve throughput.

---

## Using Poor Partition Keys

Uneven Partition Keys create hot shards.

---

## Ignoring Consumer Lag

High Iterator Age indicates processing delays.

---

## Not Using Checkpointing

Without checkpoints,

consumers may reprocess records after failures.

---

## Assuming Only One Consumer

Kinesis is designed for multiple independent consumers.

Leverage this capability for analytics, monitoring, and event processing.

---

# Summary

Producers and consumers form the foundation of Amazon Kinesis Data Streams. Producers continuously write streaming records using APIs such as `PutRecord` and `PutRecords`, while consumers retrieve and process those records using shard iterators and `GetRecords`. With support for multiple independent consumers, shared or enhanced fan-out, checkpointing, and low-latency processing, Kinesis enables scalable real-time data pipelines for analytics, monitoring, fraud detection, and event-driven applications.
