# Sticky Sessions (Session Affinity) in AWS ELB

## Definition

Sticky Sessions allow a user to continue communicating with the same backend EC2 instance during multiple requests.

The load balancer creates a binding between:
- Client
- Session
- Specific target instance

This is also called:
- Session Affinity
- Sticky Sessions

---

# Why Sticky Sessions Are Used

Useful when applications store:
- User login session
- Shopping cart data
- Temporary user state

on a specific EC2 instance memory.

Without sticky sessions:
- User requests may go to different instances
- Session data may be lost

---

# How Sticky Sessions Work

The load balancer sends a cookie to the client.

Future requests containing that cookie are forwarded to the same target instance.

---

# Types of Cookies

## 1. Duration-Based Cookies

Generated by:
- ELB itself

Example:
- AWSALB
- AWSELB

You configure:
- Cookie expiration time

---

## 2. Application-Based Cookies

Generated by:
- Your application

Example:
- SessionID
- UserToken

Useful when the application already manages sessions.

---

# Important Notes

- Supported mainly with ALB and CLB
- Sticky sessions can cause uneven traffic distribution
- Not ideal for stateless microservices architecture

---

# Best Practice

Modern cloud-native applications prefer:
- Stateless applications
- Shared session storage (Redis, DynamoDB, ElastiCache)

instead of sticky sessions.

---

# Interview Notes

## Common Interview Question

### Why are sticky sessions sometimes avoided?

Because:
- They reduce load balancing efficiency
- They can overload specific instances
- They make scaling harder
