# 08- Lambda Function URLs

# Lambda Function URLs

AWS Lambda Function URLs provide a simple way to expose a Lambda function over HTTPS without using Amazon API Gateway or an Application Load Balancer (ALB).

Every Function URL is assigned a unique HTTPS endpoint that can be invoked directly from the internet or from internal applications.

Function URLs are ideal for lightweight HTTP endpoints, webhooks, prototypes, and internal tools where advanced API management features are unnecessary.

---

# What is a Lambda Function URL?

A Function URL is a dedicated HTTPS endpoint automatically generated by AWS for a Lambda function.

Instead of:

```text
Client

↓

API Gateway

↓

Lambda
```

You can use:

```text
Client

↓

Lambda Function URL

↓

Lambda
```

AWS automatically provisions the HTTPS endpoint.

---

# High-Level Architecture

```text
Internet

↓

HTTPS Endpoint

↓

Lambda Function URL

↓

AWS Lambda

↓

Business Logic
```

Unlike API Gateway, there are no additional routing or API management layers.

---

# Example URL

A Function URL looks like:

```text
https://abc123.lambda-url.us-east-1.on.aws/
```

Every Lambda function has its own unique URL.

---

# How Function URLs Work

Request flow:

```text
Browser

↓

HTTPS Request

↓

Lambda Function URL

↓

Lambda

↓

Response

↓

Browser
```

AWS automatically invokes the Lambda function whenever the endpoint receives an HTTP request.

Lambda receives an event similar to API Gateway.

Example:

```json
{
    "version":"2.0",
    "routeKey":"$default",
    "rawPath":"/users",
    "rawQueryString":"page=1",
    "headers":{
        "host":"..."
    },
    "requestContext":{},
    "body":"..."
}
```

Your handler processes this event normally.

# Authentication Modes

Function URLs support two authorization types.

---

## NONE

Anyone with the URL can invoke the function.

```text
Internet

↓

Function URL

↓

Lambda
```

No authentication is performed.

Useful for:

- Public webhooks
- Public APIs
- Demo applications
- Temporary testing

---

## AWS_IAM

Requests must be signed using AWS Signature Version 4 (SigV4).

Architecture:

```text
IAM User

↓

Signed Request

↓

Function URL

↓

Lambda
```

Only authenticated AWS principals with the correct IAM permissions can invoke the function.

---

# Resource-Based Policy

Lambda Function URLs use a resource-based policy to control access.

Example:

```json
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "lambda:InvokeFunctionUrl"
}
```

For production workloads, avoid unrestricted public access unless explicitly required.

---

# CORS Support

Function URLs include built-in CORS configuration.

Example:

```text
Allowed Origins

https://example.com
```

Supported settings:

- Allowed Origins
- Allowed Methods
- Allowed Headers
- Allow Credentials
- Max Age

Useful for browser-based applications.

# Security Considerations

When using Function URLs:

✅ Prefer `AWS_IAM` for internal services.

✅ Restrict permissions using IAM policies.

✅ Validate all incoming input.

✅ Implement rate limiting externally if required.

❌ Do not expose sensitive administrative operations publicly without authentication.

# Limitations

Function URLs intentionally provide a minimal feature set.

Missing features include:

- API Keys
- Usage Plans
- Request Validation
- Lambda Authorizers
- JWT Authorizers
- WebSocket APIs
- Stage Variables
- Request/Response Mapping
- API Versioning
- Caching

# Common Mistakes

## Exposing Administrative APIs Publicly

Avoid:

```text
NONE Authentication

↓

Delete Customer Endpoint
```

Instead:

```text
AWS_IAM

↓

Authenticated Request

↓

Lambda
```

---

## Using Function URLs for Large Public APIs

For enterprise REST APIs requiring:

- Authentication
- Versioning
- Usage Plans
- API Keys
- Request Validation

Use API Gateway instead.

---

## Ignoring Input Validation

Never trust client input.

Always validate:

- Headers
- Query Parameters
- Request Body
- Authentication Context

# Best Practices

✅ Use Function URLs for lightweight services.

✅ Prefer `AWS_IAM` whenever possible.

✅ Enable CORS only for trusted origins.

✅ Log incoming requests.

✅ Return proper HTTP status codes.

✅ Keep functions stateless.

---

# Senior Backend Engineering Perspective

Lambda Function URLs are intentionally lightweight. They remove the operational overhead of API Gateway for simple HTTP endpoints while still benefiting from Lambda's scalability and availability.

However, they are **not** a replacement for API Gateway in enterprise environments. Production APIs typically require features such as authentication, request validation, throttling, versioning, and usage plans—areas where API Gateway excels.

A senior backend engineer should select Function URLs only when their simplicity aligns with the application's requirements, such as internal tools, webhooks, prototypes, or low-complexity services.

# Key Takeaways

- Lambda Function URLs provide a simple HTTPS endpoint for invoking Lambda functions directly.
- They support two authorization modes: `NONE` and `AWS_IAM`.
- Function URLs are well suited for lightweight APIs, webhooks, and internal services.
- They intentionally omit advanced API management capabilities such as API keys, request validation, usage plans, and custom authorizers.
- Choosing between Function URLs, API Gateway, and ALB depends on the application's security, routing, and operational requirements.
