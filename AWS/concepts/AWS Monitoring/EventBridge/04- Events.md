# Events

An **Event** is the fundamental building block of Amazon EventBridge. Every action, state change, or occurrence within an AWS service, custom application, or SaaS application is represented as an event.

Whenever something important happens, EventBridge receives an event and routes it to the appropriate targets based on configured rules.

------------------------------------------------------------------------

# What is an Event?

An **event** is a JSON document that describes something that has already happened.

Events are immutable, meaning they cannot be changed after they are created.

Examples include:

- An EC2 instance starts
- A file is uploaded to Amazon S3
- A Lambda function completes
- A payment is processed
- A customer places an order
- A user logs into an application

Think of an event as a notification saying:

> "Something happened."

------------------------------------------------------------------------

# Event Structure

Every EventBridge event is represented as a JSON object.

A simplified example:

```json
{
  "version": "0",
  "id": "12345678-1234-5678-9012-abcdef123456",
  "detail-type": "Order Created",
  "source": "com.mycompany.orders",
  "account": "123456789012",
  "time": "2026-07-05T10:15:00Z",
  "region": "ap-south-1",
  "resources": [],
  "detail": {
    "orderId": "1001",
    "customer": "John Doe",
    "amount": 250
  }
}
```

------------------------------------------------------------------------

# Event Fields

Every event contains several standard fields.

## Version

Indicates the event schema version.

Example:

```text
version = 0
```

------------------------------------------------------------------------

## ID

A globally unique identifier (UUID) for the event.

Example:

```text
id = 12345678-1234-5678-9012-abcdef123456
```

This helps uniquely identify each event.

------------------------------------------------------------------------

## Source

Identifies where the event originated.

Examples:

```text
aws.ec2

aws.s3

aws.lambda

com.mycompany.orders
```

The `source` field is commonly used in Event Patterns.

------------------------------------------------------------------------

## Detail-Type

Describes the type of event.

Examples:

```text
EC2 Instance State-change Notification

Object Created

Order Created

Payment Successful
```

This field tells consumers what kind of event occurred.

------------------------------------------------------------------------

## Account

Specifies the AWS account that generated the event.

Example:

```text
123456789012
```

Useful in multi-account environments.

------------------------------------------------------------------------

## Time

Indicates when the event occurred.

Example:

```text
2026-07-05T10:15:00Z
```

The timestamp follows the ISO 8601 format.

------------------------------------------------------------------------

## Region

Identifies the AWS Region where the event originated.

Example:

```text
ap-south-1
```

------------------------------------------------------------------------

## Resources

Contains one or more AWS resource ARNs related to the event.

Example:

```text
arn:aws:s3:::my-bucket
```

Some events may contain multiple resources, while others may leave this field empty.

------------------------------------------------------------------------

## Detail

The **detail** field contains the event-specific information.

Example:

```json
"detail": {
    "bucket": "my-images",
    "object": "profile.jpg"
}
```

This is the most important part of the event because it contains the business data.

------------------------------------------------------------------------

# Types of Events

Amazon EventBridge supports several event types.

## AWS Service Events

Generated automatically by AWS services.

Examples:

- Amazon EC2
- Amazon S3
- AWS Lambda
- AWS CloudTrail
- Amazon ECS

Example:

```text
EC2 Instance State Change
```

------------------------------------------------------------------------

## Custom Events

Generated by your own applications using the **PutEvents API**.

Example:

```text
Order Created

Payment Completed

User Registered
```

These events enable communication between microservices.

------------------------------------------------------------------------

## Partner Events

Generated by integrated SaaS applications.

Examples:

- Datadog
- Zendesk
- Shopify
- Auth0
- PagerDuty

These events allow third-party applications to integrate with AWS.

------------------------------------------------------------------------

# Event Lifecycle

The lifecycle of an event is straightforward.

```text
Event Generated

↓

Event Sent to EventBridge

↓

Rule Evaluation

↓

Pattern Match

↓

Target Invoked

↓

Optional Archive
```

If no rule matches the event, it is discarded after evaluation.

# Event Example

Suppose a customer places an order.

```json
{
  "source": "com.store.orders",
  "detail-type": "Order Created",
  "detail": {
    "orderId": "5001",
    "customer": "Alice",
    "amount": 799
  }
}
```

EventBridge evaluates this event and routes it to:

- Payment Service
- Inventory Service
- Email Service
- Analytics Service

Each service processes the event independently.

------------------------------------------------------------------------

# Event Size Limits

Some important limits to remember:

- Maximum event size: **256 KB**
- Events must be valid JSON.
- Event payloads should contain only the required information.

If the payload is larger than 256 KB, consider storing the data in Amazon S3 and sending only a reference (such as the S3 object URL or key) in the event.

------------------------------------------------------------------------

# Event Design Best Practices

When designing events:

- Keep events small.
- Use meaningful event names.
- Include only relevant business data.
- Avoid sensitive information whenever possible.
- Make events self-describing.
- Keep event schemas consistent.
- Version schemas when making breaking changes.

Well-designed events are easier to maintain and integrate across services.

------------------------------------------------------------------------

# Real-World Example

A photo-sharing application uploads images to Amazon S3.

1. A user uploads a photo.
2. Amazon S3 generates an **Object Created** event.
3. EventBridge receives the event.
4. A rule matches all image uploads.
5. AWS Lambda creates thumbnails.
6. Another Lambda updates the image database.
7. Amazon SNS sends a notification to the user.

All of these actions occur automatically without the services communicating directly.

------------------------------------------------------------------------

# Common Mistakes

Avoid these common mistakes:

- Creating oversized events.
- Including unnecessary data.
- Using inconsistent event names.
- Embedding sensitive information in events.
- Modifying event structures without versioning.

------------------------------------------------------------------------

# Key Takeaways

- Events are the foundation of Amazon EventBridge.
- Every event is represented as a JSON document.
- Events contain metadata such as `source`, `detail-type`, `time`, and `region`.
- The `detail` field contains the business-specific information.
- EventBridge supports AWS service events, custom events, and partner events.
- Well-designed events lead to scalable, maintainable, and loosely coupled event-driven architectures.
