# Log Streams

A **Log Stream** is a sequence of log events that originate from a single source within a CloudWatch Log Group. Every log event is written to exactly one Log Stream.

Log Streams help organize logs from individual resources such as EC2 instances, Lambda execution environments, or containers.

---

# What is a Log Stream?

A Log Stream is a continuous flow of log events generated by a single resource.

Hierarchy:

```text
CloudWatch Logs
      |
      v
Log Group
      |
      +-------------------+
      |                   |
      v                   v
 Log Stream A        Log Stream B
      |
      v
  Log Events
```

Each Log Group can contain multiple Log Streams.

---

# Why Use Log Streams?

Log Streams help you:

- Separate logs from different resources
- Troubleshoot individual instances
- Analyze execution history
- Search logs efficiently
- Organize logs within a Log Group

---

# Log Events

A Log Stream contains **Log Events**.

Each log event includes:

- Timestamp
- Log message
- Ingestion time

Example:

```text
2026-07-05 10:15:01  Application Started
2026-07-05 10:15:08  Connected to Database
2026-07-05 10:15:12  User Login Successful
```

Log events are stored in chronological order.

---

# Lambda Log Streams

AWS Lambda automatically creates Log Streams.

Typical structure:

```text
Log Group

/aws/lambda/payment-service

        |

        +---------------------------+
        |                           |
        v                           v

2026/07/05/[$LATEST]abc123

2026/07/05/[$LATEST]xyz789
```

Each execution environment writes to its own Log Stream.

---

# EC2 Log Streams

For EC2 instances, the CloudWatch Agent sends logs to a Log Stream.

Example:

```text
Log Group

/application/web-server

        |

        +-------------------+
        |                   |
        v                   v

i-0abc123def456

i-0xyz987uvw654
```

Each instance can have one or more Log Streams depending on the configuration.

---

# Containers and Log Streams

For Amazon ECS and Amazon EKS, containers can write logs to separate Log Streams.

This makes it easier to troubleshoot individual tasks or pods without mixing logs from other workloads.

---

# Sequence Tokens

Historically, CloudWatch Logs used **sequence tokens** to ensure log events were written in order.

Today, the AWS service automatically manages sequence handling for most SDKs and applications, so developers rarely need to manage sequence tokens manually.

---

# Real-World Example

An e-commerce application has three EC2 instances behind an Application Load Balancer.

All instances write to the same Log Group:

```text
/application/ecommerce
```

Each instance has its own Log Stream:

- i-01a23bc45de67890
- i-02b34cd56ef78901
- i-03c45de67fg89012

If one server begins returning errors, engineers can inspect only that instance's Log Stream instead of searching through logs from every server.

---

# Best Practices

- Use one Log Stream per resource or execution environment.
- Use meaningful stream names whenever creating custom streams.
- Avoid mixing logs from unrelated resources.
- Configure retention policies at the Log Group level.
- Use CloudWatch Logs Insights to search across multiple Log Streams.

---

# Key Takeaways

- A Log Stream is a sequence of log events from a single source.
- Every Log Stream belongs to exactly one Log Group.
- Lambda, EC2, ECS, and EKS commonly create separate Log Streams.
- Separating logs into streams makes troubleshooting and monitoring much easier.
