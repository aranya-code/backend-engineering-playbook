# Traces

A **trace** represents the complete journey of a single request as it travels through your distributed application.

A trace begins when a request enters your application and ends when a response is returned to the client. It contains all the information collected from every service that participates in processing the request.

AWS X-Ray uses traces to help developers understand how requests move through different application components.

------------------------------------------------------------------------

# What is a Trace?

A trace is a collection of one or more **segments** generated by different services that handle a request.

Every trace has a unique **Trace ID** that links all related segments together.

Example:

```text
Client

↓

API Gateway

↓

Lambda

↓

DynamoDB

↓

Response
```

All of these operations belong to a single trace.

------------------------------------------------------------------------

# Trace ID

Every trace generated by X-Ray has a globally unique Trace ID.

Example:

```text
1-65a8b6ef-1234567890abcdef12345678
```

A Trace ID consists of:

- Version number
- Timestamp
- Unique identifier

This ID allows AWS X-Ray to correlate requests across multiple services.

------------------------------------------------------------------------

# Trace Lifecycle

A trace follows these stages:

```text
Client Request

↓

Trace Created

↓

Segments Generated

↓

Subsegments Generated

↓

Trace Sent to X-Ray

↓

Stored and Processed

↓

Visualized in X-Ray Console
```

------------------------------------------------------------------------

# Information Stored in a Trace

A trace can contain:

- Trace ID
- Service names
- Request path
- Response time
- Latency
- Errors
- Faults
- Throttles
- HTTP information
- AWS resource details
- Annotations
- Metadata

------------------------------------------------------------------------

# Trace Timeline

The X-Ray console displays a timeline showing how much time each service spends processing the request.

Example:

```text
API Gateway      15 ms

Lambda          120 ms

DynamoDB         35 ms

External API    420 ms
```

This makes it easy to identify slow services.

------------------------------------------------------------------------

# Trace Filtering

AWS X-Ray allows filtering traces using:

- Service name
- Response time
- HTTP status code
- Annotation values
- Errors
- Faults
- Time range

Filtering helps locate problematic requests quickly.

------------------------------------------------------------------------

# Viewing Traces

You can view traces in the X-Ray console.

For each trace you can inspect:

- Timeline
- Segments
- Subsegments
- Metadata
- Exceptions
- Annotations
- Request details

------------------------------------------------------------------------

# Real-World Example

An online banking application processes a fund transfer.

```text
Customer

↓

API Gateway

↓

Transfer Service

↓

Fraud Detection

↓

Amazon RDS

↓

Notification Service
```

The complete journey is stored as a single trace.

If the Fraud Detection service is slow, the trace immediately highlights the delay.

------------------------------------------------------------------------

# Best Practices

- Enable tracing for all critical services.
- Use annotations for searchable attributes.
- Monitor slow traces regularly.
- Review traces after production incidents.
- Keep traces lightweight using sampling.

------------------------------------------------------------------------

# Key Takeaways

- A trace represents one complete request.
- Every trace has a unique Trace ID.
- Traces contain segments and subsegments.
- X-Ray visualizes traces using timelines and service maps.
- Traces are the foundation of distributed tracing.
