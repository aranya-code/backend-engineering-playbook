# Multi-Factor Authentication (MFA)

## What MFA Adds

Requires a second factor beyond a password — something the user *has* (a device generating a code) rather than just something they *know* — meaning a leaked password alone is no longer sufficient to authenticate as that identity.

## MFA Device Types on AWS

| Type | How It Works | Recommended For |
|---|---|---|
| Virtual MFA (authenticator app) | Time-based one-time password (TOTP) generated by an app (e.g., Google Authenticator, Authy) | Default recommendation for most users |
| Hardware MFA device | A physical token generating codes | High-sensitivity accounts, root user |
| FIDO Security Key | A physical USB/NFC key using public-key cryptography | Strongest phishing resistance available |
| SMS-based (legacy, discouraged) | A code sent via text message | Avoid where possible — vulnerable to SIM-swapping attacks |

## Enforcing MFA via Policy, Not Just Convention

Enabling MFA on an account doesn't force its use for every action — enforcement requires an explicit IAM policy condition:

```json
{
  "Effect": "Deny",
  "NotAction": "iam:*",
  "Resource": "*",
  "Condition": {
    "BoolIfExists": {
      "aws:MultiFactorAuthPresent": "false"
    }
  }
}
```

This denies nearly all actions unless the request was made with an MFA-authenticated session — turning "we recommend MFA" into "MFA is actually required for anything sensitive," which is the meaningful difference between a suggestion and an enforced control.

## Where MFA Matters Most

- **The root account** — non-negotiable; a compromised root account with no MFA has no ceiling on the damage possible
- **Any human identity with elevated/administrative permissions**
- **Sensitive actions specifically** — some organizations require MFA only for destructive or high-risk actions (deleting resources, modifying IAM) rather than blanket-requiring it everywhere, via targeted `Condition` blocks

## Senior-Level Considerations

- SMS-based MFA is meaningfully weaker than app-based or hardware MFA due to SIM-swapping attacks, and AWS itself recommends against it where a stronger option is available
- `aws:MultiFactorAuthPresent` only reflects whether MFA was present at the time credentials were obtained — for temporary credentials from an assumed role, this means the *original* authentication (before assuming the role) needs to have included MFA; assuming a role itself doesn't add a fresh MFA check unless explicitly configured to require it
- FIDO security keys are increasingly the recommended standard for the highest-sensitivity accounts specifically because they're resistant to phishing in a way that a TOTP code (which a user could be tricked into entering on a fake site) fundamentally isn't
