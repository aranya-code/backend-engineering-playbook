# What is a Pod?

A **Pod** is the **smallest deployable unit** in Kubernetes.

It is the smallest Kubernetes object that can be created, scheduled, and managed.

A Pod encapsulates one or more containers that share:

- Network
- Storage
- Lifecycle

Most commonly,

```
1 Pod = 1 Container
```

although Kubernetes supports multiple containers inside a single Pod.

---

# Why Do We Need Pods?

Kubernetes does not deploy containers directly.

Instead, it deploys Pods.

Reasons:

- Provides abstraction over containers
- Allows multiple containers to communicate efficiently
- Shares networking and storage
- Simplifies scheduling
- Enables self-healing

# Pod Lifecycle

Every Pod goes through several phases during its lifetime.

```
Pending

↓

Running

↓

Succeeded

↓

Failed
```

Sometimes

```
Unknown
```

---

## Pending

The Pod has been accepted by Kubernetes but is waiting for one or more resources.

Possible reasons

- Image is downloading
- Waiting for scheduling
- Persistent Volume unavailable

---

## Running

The Pod has been successfully scheduled.

At least one container is running.

Example

```
STATUS

Running
```

---

## Succeeded

All containers completed successfully.

Usually seen in Jobs.

---

## Failed

One or more containers terminated unexpectedly.

Example

```
Application Crash

↓

Failed
```

---

## Unknown

The Kubernetes Control Plane cannot determine the Pod's state.

Usually caused by node communication issues.

# Pod Networking

Every Pod gets

- Its own IP Address
- Shared localhost
- Shared network namespace

Example

```
Node

├── Pod A
│     10.244.1.10
│
├── Pod B
│     10.244.1.11
│
└── Pod C
      10.244.1.12
```

Pods communicate directly using IP addresses.

# Single Container Pod

Most applications use one container per Pod.

Example

```
Pod

↓

Nginx Container
```

Benefits

- Simplicity
- Easy scaling
- Easier debugging

# Multi-Container Pod

A Pod may contain multiple containers.

Containers work together as one application.

```
Pod

├── Main Application

└── Helper Container
```

Examples

- Logging Agent
- Metrics Collector
- Proxy

---

# Sidecar Pattern

A Sidecar container extends the functionality of the main application.

Example

```
Pod

├── Django API

└── Fluent Bit
```

Fluent Bit collects logs generated by the Django application.

Common Sidecars

- Fluent Bit
- Promtail
- Envoy
- Istio Proxy

---

# Ambassador Pattern

A helper container acts as a proxy between the application and external services.

```
Pod

├── Application

└── Proxy
```

Example

Application

↓

Envoy Proxy

↓

External API

# Adapter Pattern

An Adapter container transforms output from the main application.

Example

```
Pod

├── Application

```

# Ephemeral Containers

Ephemeral Containers are temporary containers used for debugging.

They are not restarted automatically.

Example

```bash
kubectl debug mypod -it --image=busybox
```

Useful when a running Pod needs investigation.

---

# Restart Policies

Kubernetes supports three restart policies.

## Always

Default policy.

Restart whenever a container exits.

---

## OnFailure

Restart only if the container exits with an error.

---

## Never

Never restart the container.

Typically used for Jobs.

---

# Pod Probes

Kubernetes uses probes to monitor container health.

```
Liveness Probe

↓

Readiness Probe

↓

Startup Probe
```

---

## Liveness Probe

Checks whether the application is alive.

If it fails,

```
Container

↓

Restart
```

Example

```
Health Check

↓

Failed

↓

Restart Container
```

---

## Readiness Probe

Determines whether a Pod is ready to receive traffic.

If the readiness probe fails,

- Pod is **not restarted**
- Pod is removed from Service endpoints

---

## Startup Probe

Used for slow-starting applications.

Until it succeeds,

Liveness and Readiness probes are ignored.

Useful for

- Spring Boot
- Large Java applications
- Machine Learning applications

# Resource Requests

Requests specify the minimum resources required.

Example

```yaml
resources:

  requests:

    cpu: "500m"

    memory: "256Mi"
```

---

# Resource Limits

Limits specify the maximum resources a container may use.

Example

```yaml
resources:

  limits:

    cpu: "1"

    memory: "512Mi"
```

---

# QoS Classes

Pods are assigned Quality of Service classes.

## Guaranteed

Requests = Limits

Highest priority.

---

## Burstable

Requests < Limits

Most applications use this.

---

## BestEffort

No requests or limits.

Lowest priority.

# Pod YAML Example

```yaml
apiVersion: v1

  name: nginx-pod

  labels:
    app: nginx

  containers:
```

Create the Pod

```bash
kubectl apply -f pod.yaml
```

|----------|---------|

# Common Troubleshooting

## Pod Stuck in Pending

Possible causes

- Insufficient CPU
- Insufficient Memory
- Missing Persistent Volume
- Node unavailable

---

## ImagePullBackOff

Possible causes

- Wrong image name
- Private registry authentication
- Network issues

---

## CrashLoopBackOff

Possible causes

- Application crash
- Configuration error
- Missing environment variables
- Incorrect startup command

---

## OOMKilled

Cause

Application exceeded its memory limit.

Increase the memory limit or optimize the application.

# Best Practices

✔ Usually run **one application container per Pod**

✔ Set CPU and memory requests and limits.

✔ Use Readiness and Liveness probes.

✔ Keep Pods stateless whenever possible.

✔ Use Deployments instead of creating Pods directly in production.

✔ Use ConfigMaps and Secrets for configuration.

✔ Avoid running containers as the root user.

# Interview Questions

## What is a Pod?

A Pod is the smallest deployable unit in Kubernetes that encapsulates one or more containers sharing networking and storage.

---

## Why does Kubernetes use Pods instead of containers?

Pods provide an abstraction layer that enables shared networking, storage, scheduling, and lifecycle management for containers.

---

## Can a Pod have multiple containers?

Yes. Multiple tightly coupled containers can run within a single Pod and share the same network namespace and storage.

---

## Difference between Liveness and Readiness Probes?

- **Liveness Probe:** Determines if a container is alive. If it fails, Kubernetes restarts the container.
- **Readiness Probe:** Determines if a container is ready to receive traffic. If it fails, the Pod is removed from Service endpoints but is not restarted.

---

## What is an Init Container?

An Init Container runs before the application containers to perform initialization tasks such as downloading configuration files or running database migrations.

---

## What is an Ephemeral Container?

An Ephemeral Container is a temporary debugging container that can be attached to a running Pod without modifying its main containers.

---

## Why shouldn't Pods be created directly in production?

Pods do not provide self-healing, scaling, or rolling updates. In production, Pods should be managed by higher-level controllers such as Deployments or StatefulSets.

---

# Key Takeaways

- A Pod is the smallest deployable Kubernetes object.
- Pods encapsulate one or more containers that share networking and storage.
- Most Pods contain a single application container.
- Every Pod has its own IP address.
- Init Containers prepare the environment before the application starts.
- Sidecar, Ambassador, and Adapter are common multi-container design patterns.
- Liveness, Readiness, and Startup probes help Kubernetes monitor application health.
- Resource requests and limits manage CPU and memory allocation.
- Pods should generally be managed by Deployments or StatefulSets rather than created directly.
