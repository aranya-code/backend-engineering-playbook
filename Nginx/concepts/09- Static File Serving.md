# Overview

One of the primary responsibilities of Nginx is serving **static files**.

A static file is any file that is returned to the client **without being generated by an application**.

Examples include:

- HTML
- CSS
- JavaScript
- Images
- Fonts
- Videos
- PDFs
- ZIP files

Unlike dynamic content generated by Django, FastAPI, or Node.js, static files already exist on disk.

Nginx is highly optimized for serving these files quickly and efficiently.

---

# What are Static Files?

Static files are files whose contents do not change for every request.

For example:

```
logo.png
```

Every user receives the exact same image.

Similarly,

```
style.css
```

contains the same CSS rules regardless of who requests it.

---

# Static vs Dynamic Content

| Static Content | Dynamic Content |
|---------------|-----------------|
| Already exists on disk | Generated by application |
| Served directly by Nginx | Generated by Django, FastAPI, Node.js |
| Very fast | Slower than static files |
| Low CPU usage | Higher CPU usage |
| No database required | Often requires a database |

---

# Why Nginx Excels at Serving Static Files

Nginx serves static files directly from the file system.

It does not need to:

- Execute Python code
- Create application objects
- Query a database
- Render templates

This results in:

- Lower latency
- Lower CPU usage
- Reduced memory consumption
- Higher throughput

# Basic Configuration

The simplest configuration looks like this:

```nginx
server {

    root /var/www/html;

    location / {

        index index.html;

    }

}
```

Request:

```
GET /
```

Nginx returns:

```
/var/www/html/index.html
```

---

# How File Resolution Works

Suppose the configuration is:

```nginx
server {

    root /var/www/html;

}
```

Client request:

```
GET /about.html
```

Nginx searches for:

```
/var/www/html/about.html
```

If the file exists, it is returned immediately.

```

Request:

```

```

Nginx searches for:

```
/var/www/images/logo.png

If found, the image is returned.

└── assets

```

```

Returned file:

```

```

# Understanding the `root` Directive

The `root` directive defines the base directory where Nginx begins searching for files.

Example:

```nginx
root /usr/share/nginx/html;
```

Request:

```
/index.html
```

Resolved path:

```
/usr/share/nginx/html/index.html
```

Request:

```
/css/style.css
```

Resolved path:

```
/usr/share/nginx/html/css/style.css
```

---

# Understanding the `index` Directive

When a client requests a directory instead of a file, Nginx looks for the file specified by the `index` directive.

Example:

```nginx
index index.html;
```

Request:

```
GET /
```

Nginx searches for:

```
index.html
```

Multiple index files can also be configured.

```nginx
index index.html index.htm default.html;
```

Nginx returns the first file that exists.

---

# The `try_files` Directive

`try_files` allows Nginx to check whether one or more files exist before deciding what to do.

Example:

```nginx
location / {

    try_files $uri $uri/ =404;

}
```

Processing:

1. Check if the requested file exists.
2. Check if a matching directory exists.
3. Return **404 Not Found** if neither exists.

---

# Single Page Applications (React, Vue, Angular)

Modern frontend frameworks often require every unknown route to return `index.html`.

Example:

```nginx
location / {

    try_files $uri $uri/ /index.html;

}
```

Request:

```
/dashboard/profile
```

If the file does not exist:

```
index.html
```

is returned, allowing the frontend router to handle navigation.

# Django Example

Typical project:

```text
project/

├── static/

│     ├── css

│     ├── js

│     └── images

└── media/
```

Nginx configuration:

```nginx
location /static {

    root /var/www/project;

}
```

Request:

```
GET /static/css/style.css
```

Resolved path:

```
/var/www/project/static/css/style.css
```

The Django application is **not** involved in serving this file.

# Best Practices

- Let Nginx serve static files instead of the application server.
- Keep static assets in dedicated directories.
- Use meaningful directory structures.
- Use `try_files` to avoid unnecessary backend requests.
- Enable caching for static assets in production.
- Use compression for CSS, JavaScript, and HTML files.

---

# Common Mistakes

## Incorrect `root` Path

A wrong root directory causes **404 Not Found** errors because Nginx searches in the wrong location.

---

## Serving Static Files Through the Application

Requests for images, CSS, and JavaScript should normally be handled directly by Nginx instead of Django or FastAPI.

This reduces load on the application server.

---

## Forgetting `index`

Without an `index` directive, requesting a directory may result in an error instead of serving the default page.

---

## Misunderstanding `try_files`

`try_files` checks the filesystem in order.

The first existing file is served.

If none exist, the final fallback is used.

# Key Takeaways

- Static files are served directly from the filesystem without involving the application.
- The `root` directive defines the base directory for file lookup.
- The `index` directive specifies the default file for directory requests.
- `try_files` provides flexible file lookup and fallback behavior.
- Serving static files through Nginx improves application performance, scalability, and resource utilization.
