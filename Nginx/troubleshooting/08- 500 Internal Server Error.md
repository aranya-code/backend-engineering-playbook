# Overview

The **500 Internal Server Error** indicates that the request successfully reached the server, but an unexpected error occurred while processing it.

Unlike **502 Bad Gateway** or **504 Gateway Timeout**, a **500** error usually originates from the **backend application**, not from Nginx itself.

In most production environments, Nginx simply forwards the response generated by the application server (such as Django, FastAPI, PHP-FPM, or Node.js).

Common causes include:

- Application exceptions
- Database connection failures
- Incorrect application configuration
- Missing environment variables
- File permission issues
- Programming errors
- Application startup failures

# Why This Happens

A **500 Internal Server Error** means the backend application failed while processing the request.

Typical flow:

```text
Browser

↓

Nginx

↓

Application

↓

Unhandled Exception

↓

HTTP 500
```

Nginx simply returns the response generated by the backend.

# Common Causes

| Cause | Description |
|--------|-------------|
| Application bug | Unhandled exception |
| Database failure | Connection refused |
| Missing environment variable | Configuration error |
| Missing dependency | Python package not installed |
| Permission issue | Cannot access required file |
| Failed migration | Database schema mismatch |
| Programming error | Syntax or runtime error |

# Error Message Decoder

## Python Exception

```text
ModuleNotFoundError
```

### Meaning

A required Python module is missing.

### Solution

Install the missing dependency.

---

## Database Error

```text
OperationalError
```

### Meaning

Application cannot connect to the database.

### Solution

Verify:

- Database running
- Credentials
- Network
- Environment variables

---

## Environment Variable

```text
KeyError

SECRET_KEY
```

### Meaning

Required environment variable is missing.

### Solution

Verify the application configuration.

# Step-by-Step Troubleshooting

## Step 1 — Verify Nginx

```bash
sudo nginx -t
```

Ensure Nginx configuration is valid.

---

## Step 2 — Test Backend

```bash
curl http://127.0.0.1:8000
```

If the backend returns **500**, the issue is inside the application.

---

## Step 3 — Review Application Logs

Docker

```bash
docker logs backend
```

Systemd

```bash
journalctl -u gunicorn
```

Look for:

- Exceptions
- Tracebacks
- Missing modules
- Database errors

---

## Step 4 — Verify Database

Example

```bash
systemctl status postgresql
```

or

```bash
docker ps
```

Ensure the database is healthy.

---

## Step 5 — Verify Environment Variables

Example

```bash
printenv
```

Confirm:

- Database credentials
- API keys
- Secret keys
- Configuration values

---

## Step 6 — Restart Backend

```bash
systemctl restart gunicorn
```

or

```bash
docker restart backend
```

# Best Practices

- Never expose application stack traces in production.
- Log all application exceptions.
- Validate environment variables during startup.
- Use centralized logging.
- Monitor backend health endpoints.

# Real Production Scenario

## Problem

A Django deployment completed successfully.

Every request immediately returned:

```text
500 Internal Server Error
```

---

## Investigation

Nginx logs showed no errors.

Backend logs contained:

```text
OperationalError

could not connect to PostgreSQL
```

The deployment accidentally changed the database password.

---

## Resolution

The environment variable was corrected.

The application was restarted.

Requests immediately returned:

```text
200 OK
```

# Verification Checklist

After fixing the issue:

- ✅ Backend service is running
- ✅ Database is reachable
- ✅ Environment variables are correct
- ✅ No application exceptions
- ✅ Nginx returns HTTP 200
- ✅ Logs contain no new errors

# Common Mistakes

❌ Assuming every 500 error is caused by Nginx

❌ Ignoring backend logs

❌ Restarting Nginx instead of the application

❌ Deploying without validating environment variables

❌ Not checking database connectivity

---

# Prevention Tips

- Add health check endpoints.
- Validate configuration during application startup.
- Use structured logging.
- Monitor application exceptions.
- Test deployments in staging before production.

---

# Production Notes

In modern architectures, **500 Internal Server Error is almost always an application issue rather than an Nginx issue**.

Before changing Nginx:

- Verify backend logs.
- Verify database connectivity.
- Verify Redis (if used).
- Verify environment variables.
- Verify application health.

Only investigate Nginx after confirming the backend is functioning correctly.
